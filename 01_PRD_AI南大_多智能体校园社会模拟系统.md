# AI 南大｜多智能体校园社会模拟系统 PRD

## 1. 产品定位

AI 南大是一个参考 HKUST Aivilization、Stanford Generative Agents 与 AI Town 的轻量级多智能体校园社会模拟原型。它把南昌大学校园抽象为一个 AI Agent 可以感知、行动、对话、记忆和反思的虚拟环境，让用户观察一批 AI 新生在校园中生活一周。

项目的第一目标是作品集和简历展示，第二目标是为社团协作开发提供清晰方向。它不是港科大 Aivilization 的规模复刻，而是对其核心范式的本土化、低成本、可演示实现。

核心命题：

> 如果把校园生活建模为地点、时间、课程、社团、关系和事件，AI Agent 能否产生足够可读、可看、可传播的校园生活叙事？

## 2. 用户与使用场景

### 2.1 目标用户

- 校内学生：希望看到“AI 新生如何体验南大”的轻量互动内容。
- 社团新成员：通过项目理解社团的 AI 产品能力和技术方向。
- 面试官或项目评审：通过文档和 demo 判断项目复杂度、技术深度和产品判断。
- 项目协作者：根据 PRD 拆分前端、后端、Agent、策划、美术和测试任务。

### 2.2 用户场景

用户打开网页后，不需要安装客户端，也不需要登录。首页直接进入一个简化的南大校园模拟界面。用户可以看到时间正在推进，AI 角色在地图上移动，事件流不断出现，例如：

- “李知远 8:10 从宿舍出发去教学楼，但因为不熟悉路线迟到了。”
- “陈念在食堂排队时遇到同班同学，两人约好下午去社团招新。”
- “周岚在图书馆捡到一卡通，触发失物招领事件。”
- “一个 AI 新生因为连续社交失败，开始反思自己是否应该主动加入社团。”

用户可以点击角色，查看角色档案、当前目标、最近记忆、关系网络和本周经历。用户也可以对少数角色输入轻量干预，例如“提醒他去参加招新”“建议她去失物招领处看看”。

## 3. 产品目标

第一版目标不是追求大规模，而是验证以下问题：

- 可看性：用户是否愿意观察 Agent 的校园生活事件流？
- 传播性：角色故事、校园梗、随机事件是否容易截图传播？
- 技术展示：Agent 行为循环是否能清楚展示人格、计划、记忆和反思？
- 成本可控：能否用规则 + 事件触发 + 少量 LLM 调用跑出可演示结果？
- 可协作：是否能拆成社团成员可以参与的策划、前端、后端、Agent 和测试任务？

## 4. MVP 范围

### 4.1 Agent 数量

MVP 默认 5-20 个 Agent。建议第一轮从 8 个开始：

- 5 个 AI 新生。
- 2 个学长学姐。
- 1 个社团负责人或校园服务角色。

这个规模足够产生关系和事件，又不会让调度、成本和内容质量失控。

### 4.2 地图与地点

第一版使用简化地图，不追求真实 GIS 精度。地点采用节点图建模：

- 宿舍区。
- 教学楼。
- 食堂。
- 图书馆。
- 体育场。
- 社团招新点。
- 失物招领点。
- 校门或商业街。

每个地点包含：

- 地点名称。
- 地点类型。
- 开放时间。
- 可触发事件。
- 当前在场 Agent。
- 与其他地点的路径距离。

### 4.3 时间与日程

系统模拟一周校园生活。时间可以加速推进，默认以 10-30 分钟为一个 tick。

每个 Agent 至少有：

- 每日基础作息。
- 课程表。
- 饭点倾向。
- 社团兴趣。
- 自由活动时间。
- 临时事件响应能力。

### 4.4 核心事件

MVP 至少包含以下事件类型：

- 上课：准时、迟到、逃课、找不到教室。
- 吃饭：排队、偶遇、拼桌、吐槽菜品。
- 社团：招新、面试、活动、分工。
- 迷路：路线选择失败，向别人求助。
- 失物招领：捡到物品、寻找物品、归还物品。
- 社交：认识新朋友、产生误会、建立关系。
- 随机事件：下雨、临时讲座、活动冲突、校园热点。

### 4.5 用户可操作内容

第一版用户不是全能上帝，而是轻量观察者和干预者：

- 查看地图和 Agent 位置。
- 查看全局事件流。
- 点击 Agent 查看详情。
- 查看某个 Agent 的记忆摘要。
- 向单个 Agent 输入一句建议或任务。
- 触发少量预设事件，例如“今晚社团招新”“食堂遗失校园卡”。

## 5. 核心玩法

### 5.1 观察校园

用户进入页面后，首先看到一个简化校园地图。地图上展示 Agent 头像、位置和当前动作。用户不需要先读说明，应该能直接理解“这些 AI 角色正在校园里生活”。

### 5.2 阅读事件流

事件流是 MVP 的核心传播载体。每个事件应短、具体、带有校园语境。

事件流示例：

- “08:05 王一诺从宿舍出发，目标是赶到高数课。”
- “08:22 王一诺在教学楼附近迷路，向林见川求助。”
- “12:14 陈念在一食堂遇到同专业同学，两人开始聊社团招新。”
- “19:40 周岚总结今天的经历，认为自己需要更主动和别人说话。”

### 5.3 查看角色

角色详情页包含：

- 基础身份：姓名、年级、专业、性格、人设。
- 当前状态：地点、目标、心情、精力、日程。
- 关系网络：认识谁、关系强度、最近互动。
- 记忆摘要：今天发生了什么、重要经历、反思。
- 行为记录：最近行动和对话。

### 5.4 轻量干预

用户可以对角色进行有限干预。干预不是直接改写结果，而是作为一条外部建议进入 Agent 记忆或目标队列。

示例：

- “你可以去社团招新点看看。”
- “你今天好像还没吃饭。”
- “如果你捡到了校园卡，可以去失物招领处。”

系统将干预写入 Agent 的短期记忆，并在下一次计划时影响行动。

## 6. Agent 行为循环

Agent 每次行动遵循以下循环：

```text
人格设定 -> 目标生成 -> 日程规划 -> 环境感知 -> 行动选择 -> 对话生成 -> 记忆写入 -> 事件反思
```

### 6.1 人格设定

每个 Agent 有稳定但可演化的人格档案：

- 姓名和身份。
- 专业与兴趣。
- 性格倾向。
- 长期目标。
- 社交偏好。
- 校园适应状态。
- 价值观或行为习惯。

MVP 不需要复杂心理模型，但要保证角色差异明显。

### 6.2 目标生成

目标分为三层：

- 长期目标：适应大学生活、加入社团、认识朋友、保持成绩。
- 日目标：今天上课、吃饭、参加招新、完成作业。
- 即时目标：去某地点、和某人说话、处理当前事件。

### 6.3 日程规划

系统每天开始时为 Agent 生成基础计划。计划不需要每一步都由 LLM 生成，可以由规则模板产生，再由 LLM 在关键节点润色或调整。

### 6.4 环境感知

Agent 在每个 tick 感知有限范围内的信息：

- 当前地点。
- 当前时间。
- 附近 Agent。
- 地点可用事件。
- 自己的日程压力。
- 近期记忆。
- 用户干预。

### 6.5 行动选择

行动优先由规则系统决定，LLM 只在冲突、对话和复杂事件中介入。

常见行动：

- move：移动到地点。
- attend_class：上课。
- eat：吃饭。
- talk：与 Agent 对话。
- join_activity：参加活动。
- ask_help：求助。
- report_lost_item：处理失物。
- reflect：反思。
- idle：等待或休息。

### 6.6 对话生成

对话只在有事件价值时触发，不做全时聊天。触发条件包括：

- 两个 Agent 在同一地点且有共同目标。
- 一方需要求助。
- 关系变化事件。
- 用户干预引发回应。
- 社团活动或失物事件需要协作。

### 6.7 记忆写入

记忆分为：

- 短期记忆：最近行动、对话、事件。
- 长期记忆：重要经历、关系变化、反思结论。
- 关系记忆：对其他 Agent 的印象。
- 用户干预记忆：用户输入的建议或任务。

### 6.8 事件反思

每天结束时，Agent 对当天重要事件做摘要：

- 今天发生了什么？
- 哪些人值得记住？
- 明天要调整什么？
- 当前目标是否改变？

反思写入长期记忆，影响后续计划。

## 7. 世界建模

### 7.1 世界状态

World State 是系统中心数据结构，包含：

- 当前模拟时间。
- 地点列表。
- Agent 列表。
- 课程和活动。
- 事件队列。
- 全局事件流。
- 关系网络。

### 7.2 地点模型

地点不是纯视觉元素，而是行动约束。每个地点决定 Agent 能做什么。

例如：

- 教学楼：上课、迟到、遇到同学。
- 食堂：吃饭、偶遇、排队。
- 图书馆：学习、安静对话、捡到物品。
- 社团招新点：报名、面试、认识学长学姐。

### 7.3 事件模型

事件包含：

- 事件 ID。
- 事件类型。
- 发生时间。
- 发生地点。
- 参与 Agent。
- 事件描述。
- 对状态的影响。
- 是否需要 LLM 生成文本。

## 8. 低成本模拟策略

MVP 的核心原则是：

> 规则驱动日常行为，LLM 处理关键节点。

### 8.1 规则优先

以下内容优先用规则完成：

- 按课表移动。
- 饭点去食堂。
- 晚上回宿舍。
- 地点开放时间。
- 路径计算。
- 基础事件触发。

### 8.2 LLM 触发条件

只有在以下场景调用 LLM：

- 生成角色人格和初始目标。
- 多 Agent 对话。
- 复杂事件决策。
- 每日反思。
- 用户干预解释。
- 重要事件的叙事化输出。

### 8.3 上下文压缩

每个 Agent 不携带完整历史，而是携带：

- 固定人设。
- 当前状态。
- 最近 5-10 条短期记忆。
- 长期记忆摘要。
- 与当前相关的关系记忆。
- 当前事件上下文。

### 8.4 缓存与模板

可缓存内容：

- 固定角色档案。
- 地点描述。
- 常见事件模板。
- 常见对话开头。
- 日程规划模板。

### 8.5 分层决策

第一层：规则判断是否需要行动。  
第二层：规则判断是否需要 LLM。  
第三层：LLM 只对具体问题输出结构化结果。  
第四层：系统校验结果是否合法，再写回世界状态。

## 9. 技术架构

### 9.1 总体架构

```text
Web Frontend
  -> Simulation API
  -> Agent Runtime
  -> World State Store
  -> Memory Store
  -> LLM Service
  -> Event Feed
```

### 9.2 前端

前端负责：

- 展示校园地图。
- 展示 Agent 位置和状态。
- 展示事件流。
- 展示角色详情。
- 提供用户干预输入。
- 展示模拟时间控制。

推荐形态：

- 轻量网页。
- 桌面端优先，移动端可浏览。
- 视觉上更像“校园模拟观察器”，不是营销落地页。

### 9.3 后端

后端负责：

- 模拟 tick 推进。
- Agent 调度。
- 世界状态更新。
- 事件生成。
- LLM 调用。
- 记忆读写。
- 成本与调用频率控制。

### 9.4 数据存储

MVP 可使用简单数据库或 JSON 存储，核心表或集合包括：

- agents。
- locations。
- schedules。
- events。
- memories。
- relationships。
- user_interventions。

### 9.5 LLM 服务层

LLM 服务层不直接暴露给前端。后端通过固定接口调用：

- generate_agent_profile。
- plan_day。
- decide_complex_action。
- generate_dialogue。
- summarize_memory。
- reflect_day。

所有 LLM 输出需要结构化校验，避免直接把不合法行动写入世界状态。

## 11. 验收标准

MVP 达到以下条件即可算第一版成立：

- 能连续模拟一周校园生活。
- 至少 5 个 Agent 有完整人设、目标、日程和记忆。
- 至少 6 个地点能触发不同事件。
- 事件流能持续输出可读校园故事。
- 至少支持一种用户干预。
- 至少有一次多 Agent 对话。
- 每个 Agent 每天能生成一条反思。
- 前端能展示地图、事件流、角色详情。
- 单次演示可在 5 分钟内说明项目价值。
- 小范围邀请 5-10 名同学体验，并记录反馈。

## 12. 风险与边界

### 12.1 技术风险

- LLM 输出不稳定，可能导致行为不连贯。
- 多 Agent 对话成本容易失控。
- 记忆过长会导致上下文膨胀。
- 角色行为可能同质化。

### 12.2 产品风险

- 用户可能只觉得“像聊天机器人换了皮”。
- 事件流如果不够具体，会缺乏传播性。
- 没有校园梗和真实细节，项目会变空。

### 12.3 安全与伦理边界

- 不采集真实学生隐私。
- 不模拟真实具体个人。
- 不允许生成攻击、歧视、骚扰或违法行为。
- 不把模拟结果包装成真实社会科学结论。
