# AI 南大 MVP 任务拆解

## 1. MVP 总目标

做出一个可演示的网页原型，让用户能看到 5-20 个 AI Agent 在南昌大学校园中生活一周。第一版重点是行为链可读、演示稳定、成本可控，而不是规模宏大。

最小成功标准：

- 有地图。
- 有 Agent。
- 有时间推进。
- 有事件流。
- 有角色详情。
- 有记忆和关系。
- 有至少一种用户干预。
- 有 LLM 参与对话、反思或复杂事件。

## 2. 模块一：前端展示

目标：

让用户一打开网页就能理解“AI 角色正在校园里生活”。

需要实现：

- 主页面布局。
- 简化校园地图。
- Agent 头像和位置展示。
- 模拟时间显示。
- 全局事件流。
- Agent 详情面板。
- 用户干预输入框。
- 开始、暂停、快进模拟按钮。

验收标准：

- 用户能看到 Agent 在不同地点变化。
- 用户能点击 Agent 查看详情。
- 用户能阅读连续事件流。
- 页面不需要说明书也能理解基本玩法。

## 3. 模块二：世界状态

目标：

把校园抽象成可被 Agent 感知和行动的环境。

需要实现的数据：

- locations：地点。
- paths：地点之间的连接和距离。
- current_time：当前模拟时间。
- agents：角色状态。
- schedules：课表和活动。
- events：事件队列和历史事件。
- relationships：角色关系。
- memories：角色记忆。

第一版地点建议：

- 宿舍区。
- 教学楼。
- 食堂。
- 图书馆。
- 体育场。
- 社团招新点。
- 失物招领处。
- 校门或商业街。

验收标准：

- 系统能根据时间推进世界状态。
- 每个 Agent 都有当前位置和当前状态。
- 事件发生后能写回世界状态。

## 4. 模块三：Agent Runtime

目标：

让 Agent 不只是随机移动，而是根据身份、目标、日程、环境和记忆做出行动。

需要实现：

- AgentProfile：姓名、专业、性格、兴趣、长期目标。
- AgentState：地点、精力、心情、当前目标、当前任务。
- Planner：生成每日计划和即时目标。
- Perception：读取当前地点、附近角色、可用事件和记忆。
- ActionSelector：选择下一步行动。
- Dialogue：在必要时生成对话。
- Reflection：每日总结和目标更新。

第一版行动类型：

- move。
- attend_class。
- eat。
- talk。
- join_activity。
- ask_help。
- handle_lost_item。
- reflect。
- idle。

验收标准：

- Agent 会按照课表去上课。
- Agent 会在饭点去食堂。
- Agent 会在事件触发时改变计划。
- Agent 的行为能解释得通。

## 5. 模块四：记忆系统

目标：

让 Agent 的过去经历影响后续行为，而不是每次都像第一次出现。

需要实现：

- 短期记忆：最近行动和对话。
- 长期记忆：每日总结和重要事件。
- 关系记忆：对其他 Agent 的印象。
- 干预记忆：用户给出的建议或任务。

记忆写入规则：

- 普通移动不一定写入长期记忆。
- 对话、迟到、社团、失物招领、关系变化写入短期记忆。
- 每天结束时用 LLM 或模板生成长期摘要。

验收标准：

- Agent 能提到之前发生过的重要事件。
- Agent 对认识过的人有不同反应。
- 用户干预能进入后续决策。

## 6. 模块五：事件系统

目标：

让校园生活产生故事，而不是只有机械移动。

事件类型：

- 课程事件。
- 饭点事件。
- 社团事件。
- 迷路事件。
- 失物招领事件。
- 社交事件。
- 天气或校园随机事件。

事件流程：

```text
触发条件 -> 创建事件 -> 选择参与 Agent -> 更新状态 -> 生成事件文本 -> 写入事件流 -> 写入相关记忆
```

验收标准：

- 一周模拟中至少出现 20 条有意义事件。
- 至少出现 3 种不同事件类型。
- 至少有 1 次事件导致 Agent 计划改变。

## 7. 模块六：LLM 调用层

目标：

控制 LLM 只在有价值的地方出现，避免成本失控。

LLM 用途：

- 生成 Agent 初始人设。
- 生成关键对话。
- 处理复杂事件选择。
- 生成每日反思。
- 将事件写成更自然的文本。

不使用 LLM 的地方：

- 普通移动。
- 常规上课。
- 常规吃饭。
- 地点开放判断。
- 简单日程推进。

接口建议：

- generate_agent_profile。
- plan_day。
- generate_dialogue。
- decide_event_action。
- summarize_memory。
- reflect_day。

验收标准：

- LLM 输出经过结构化校验。
- 单次演示的调用次数可控。
- 没有 LLM 时，系统仍可用规则跑出基础模拟。

## 8. 模块七：策划与内容

目标：

让项目有南大校园语境，而不是通用 AI 小镇换皮。

需要准备：

- 8 个初始 Agent 设定。
- 6-8 个校园地点描述。
- 一周课程表模板。
- 10 个社团活动事件。
- 10 个随机事件。
- 20 条校园语境事件文案模板。
- 角色关系初始设定。

验收标准：

- 用户能感觉这是高校校园，而不是抽象小镇。
- 事件文本具体，不空泛。
- 角色之间有差异。

## 9. 模块八：测试与演示

目标：

验证项目是否真的可看、可讲、可写进简历。

测试方式：

- 邀请 5-10 名同学体验。
- 每人看 3-5 分钟模拟。
- 记录他们记住了哪些角色和事件。
- 询问他们是否愿意截图分享。
- 询问他们是否理解这是 Agent 项目。

测试问题：

- 你觉得哪个角色最有记忆点？
- 哪条事件最像真实校园生活？
- 你有没有想继续看下去？
- 你觉得这和普通 AI 聊天有什么区别？
- 你觉得哪里无聊或看不懂？

验收标准：

- 至少 3 名用户能复述一个 Agent 的故事。
- 至少 3 名用户能说出项目和普通聊天机器人的区别。
- 至少收集 10 条有效反馈。

## 10. 推荐开发顺序

第一阶段：静态原型。

- 地图。
- 角色列表。
- 假事件流。
- 角色详情。

第二阶段：规则模拟。

- 时间推进。
- 地点移动。
- 课表和饭点。
- 规则事件。

第三阶段：Agent 机制。

- 人格。
- 目标。
- 记忆。
- 行动选择。

第四阶段：LLM 接入。

- 对话。
- 反思。
- 复杂事件。
- 事件润色。

第五阶段：演示和测试。

- 优化文案。
- 降低成本。
- 做用户测试。
- 整理作品集材料。

## 11. 最小团队分工

如果社团协作，建议最小分工：

- 产品/策划：玩法边界、角色设定、事件设计、用户测试。
- 前端：地图、角色展示、事件流、详情面板。
- 后端：模拟推进、状态存储、API。
- Agent 工程：行为循环、记忆、LLM 调用。
- 视觉/内容：地图风格、角色头像、校园文案。

如果一个人做，优先顺序是：

1. 静态网页展示。
2. 规则模拟。
3. 少量 Agent 记忆。
4. LLM 对话和反思。
5. 用户测试。
